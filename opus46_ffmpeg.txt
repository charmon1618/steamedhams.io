вқҜ What about the slices stuff looks worth investigating?

в—Ҹ The most interesting cluster is the per-slice granularity bugs вҖ” they're connected to each other and to a real correctness problem.

  The core issue: ref_poc / ref_count stored per-picture, not per-slice

  h264dec.h:140-141:
  int ref_poc[2][2][32];  ///< ... (FIXME need per slice)
  int ref_count[2][2];    ///< ... (FIXME need per slice)

  These live on H264Picture вҖ” one copy per decoded frame. But H.264 allows each slice in a frame to have its own reference picture list reordering.
   Slice A can reference frames [0,3,5], slice B can reference [1,2,4]. By storing ref_poc/ref_count on the picture, the last slice's reordering
  silently overwrites all previous slices' data.

  This matters because ref_poc is consumed later for direct motion vector prediction (B-slices) and frame reordering output. If two slices in the
  same frame use different ref list orders, the wrong POCs get used for temporal direct prediction on slices decoded earlier.

  Why it hasn't exploded yet

  Most real-world H.264 encodes use one slice per frame, or multiple slices with identical ref list ordering. The bug only triggers with:
  - Multi-slice B-frames
  - Where slices use different ref_pic_list_modification commands

  That's rare in the wild but legal per spec, and fuzz-testable.

  The related per-slice fixmes reinforce this

  - qp_thresh (line 2323) вҖ” same pattern, different consequence. One QP threshold per slice means the deblocking filter uses stale thresholds on
  MBs that came from earlier slices with different QP. Visible as banding artifacts at slice boundaries.
  - first_mb_addr == 0 field detection (line 2088) вҖ” the heuristic for "is this a new field?" is just "is this the first MB?" That breaks if a
  field's first slice doesn't start at MB 0 (legal in error recovery / partial frames).

  What I'd look at first

  1. Trace ref_poc writes and reads. Find every place ref_poc is written (should be in ff_h264_build_ref_list or similar) and every place it's read
   (direct MV prediction, reordering). Confirm the overwrite-on-second-slice theory.
  2. Check if H264SliceContext already has the right data. sl->ref_list[2][48] exists per-slice вҖ” the per-slice ref info might already be there but
   ref_poc on the picture is a stale copy that downstream code reads instead. If so, the fix is rewiring readers to use sl->ref_list rather than
  pic->ref_poc.
  3. The MBAFF cbp_table checks (lines 2399, 2403) are worth a look too вҖ” they're in the hot decode loop and "FIXME check MBAFF" on bitmasking code
   means the interlaced deblock path could silently read wrong nonzero-coeff counts, producing visual artifacts on interlaced content.

  The 16x32 decode question (lines 2621, 2692) is a perf optimization, not a bug вҖ” lower priority.

  int truncated to uint16_t on every write. No cast, no mask вҖ” just implicit truncation.

  The type mismatch and what it causes

  slice_num is int (32-bit), slice_table entries are uint16_t (16-bit). This creates two distinct problems:

  1. Sentinel comparison is broken for deblocking mode != 2

  h264_slice.c:2349:
  if (h->slice_table[top_xy] == 0xFFFF)
      top_type = 0;

  This checks "is the neighbor uninitialized?" by comparing against 0xFFFF вҖ” the memset sentinel. This works because slice_table is uint16_t, so
  the literal 0xFFFF matches.

  But h264_slice.c:2344 (deblocking mode 2):
  if (h->slice_table[top_xy] != sl->slice_num)
      top_type = 0;

  Here slice_table[top_xy] is promoted to int for the comparison. If slice_num is, say, 5, and the neighbor is uninitialized (0xFFFF = 65535 as
  int), the comparison correctly says "not same slice." So the sentinel case is implicitly handled... but only by accident.

  2. Wrap-around at 65536 slices вҖ” silent collision

  slice_num increments without bound as an int. After 65536 slices (across frames вҖ” current_slice is reset per frame, so this needs 65536+ slices
  in one frame, which is unlikely but spec-legal for huge tiled frames), the truncated uint16_t wraps to 0. Slice 65536 and slice 0 become
  indistinguishable in the table. Every neighbor check вҖ” deblocking boundary, MV prediction, CABAC context вҖ” would treat MBs from different slices
  as same-slice.

  3. The ref2frm index uses & (MAX_SLICES - 1), not the table

  h264_slice.c:1995:
  int *ref2frm = h->ref2frm[sl->slice_num & (MAX_SLICES - 1)][j];

  And the readers (h264_slice.c:2221, 2236):
  const int *ref2frm = &h->ref2frm[h->slice_table[top_xy] & (MAX_SLICES - 1)][list][...];

  MAX_SLICES is 32, so this masks to 5 bits. ref2frm has 32 slots. If there are more than 32 slices in a frame, the writer at slice 32 overwrites
  slot 0's ref2frm data, but any MB that was stamped with slice_num=0 in slice_table will now read slot 0 вҖ” which contains slice 32's ref mapping.
  Wrong reference frames for the deblocking filter.

  The warning at line 1990 even acknowledges this:
  "Possibly too many slices (%d >= %d), increase MAX_SLICES and recompile if there are artifacts"
